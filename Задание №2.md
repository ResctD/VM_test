## Часть 1. Структурный анализ
______
1. Уровни стандарт придуманы для создания профилей строгости, влияющих на бизнес-функции, аппаратную нагрузку. 
- Level 1 - базовая безопасная конфигурация, чтобы снизить поверхность атаки, перекрыв большую часть уязвимых точек узла. 
- Level 2 - строгая безопасная конфигурация, для настройки критических узлов организации. Перед эксплуатацией таких параметров рекомендуется производить предварительное функциональное тестирование и оценку производительности.

2. Поля проверок
- **№ проверки** (ID): Номер проверки
- **Наименование** (Title): Имя проверки и контекст
- **Профиль применимости** (Profile Applicability): для указания к какому уровню безопасной конфигурации относится текущая проверка
- **Описание** (Description): Объяснение что проверяется и какое значение рекомендуется как безопасное;
- **Обоснование** (Rationale): Объяснение почему этот параметр должен быть использован
- **Значение по умолчанию** (Default Value): значение, которое установлено по умолчанию
- **Влияние** (Impact): Информация по  возможным последствиям применения настройки. К каким утилитам/функциям пользователь получит ограничения, если параметр будет включен
- **Аудит** (Audit): Команды или скрипты или инструкции для проверки текущего состояния
- (**Remediation**): Инструкция по настройке параметра
- **Ссылки** (References): Адреса на документацию Microsoft или внешние источники
- **CIS Controls**: Ссылка на соответствующий контроль в CIS Controls и группу реализации.
- **Комментарий** (Note): Содержит дополнительную информацию в виде предупреждений, исключений.

Пример:
- ID: 1.1.1
- GRIP: MS-00000001
- Title: Ensure 'Enforce password history' is set to '24 or more password(s)' (Automated)
- Profile: Level 1
- Description:  Этот параметр политики определяет минимальное количество символов, которое должно содержать имя пользователя. Рекомендуемое состояние — 14 или более символов.
- **Rationale**: Увеличение минимальной длины пароля снижает вероятность успешной атаки методом подбора.
- Impact: The major impact of this configuration is that users must create a new password every time they are required to change their old one. If users are required to change their passwords to new unique values, there is an increased risk of users who write their passwords somewhere so that they do not forget them. Another risk is that users may create passwords that change incrementally (for example, password01, password02, and so on) to facilitate memorization but make them easier to guess. Also, an excessively low value for the Minimum password age setting will likely increase administrative overhead, because users who forget their passwords might ask the help desk to reset them frequently.
- Audit: Navigate to the UI Path articulated in the Remediation section and confirm it is set as prescribed.
- Remediation: To establish the recommended configuration via GP, set the following UI path to 24 or more password(s): Computer Configuration\Policies\Windows Settings\Security Settings\Account Policies\Password Policy\Enforce password history
- Default Value: 24 passwords remembered on domain members. 0 passwords remembered on standalone workstations.
- References: 
	- https://www.cisecurity.org/white-papers/cis-password-policy-guide/ 
	- https://learn.microsoft.com/en-us/windows/security/threat-protection/securitypolicy-settings/password-policy 
- CIS Controls:
	- v8, 5.2 Use Unique Passwords, IG 1, IG 2, IG 3
	- v7, 16.2 Configure Centralized Point of Authentication, IG 2, IG 3

2. Пример Проверка длины пароля
**Наименование**: Ensure 'Minimum password length' is set to '14 or more character(s)' (**Automated**) - По описанию понятно что попали в нужную проверку.
**Профиль**: Level 1 (L1) - Базовая конфигурация безопасности
**Описание**: Минимальная длинная пароля составляет от 14 символов 
**Обоснование**: Параметр направлен на противодействие атакам типа перебора паролей через словари распространённых слов, фраз и через перебор всех возможных символов
**Влияние**: Сообщают о нескольких последствиях ограничения длинны пароля пользователям:
- Длинные пароли скорее всего будут записывать и появляется риск, что хранение записей будет реализовано небезопасным способом
- Ввод длинного пароля увеличивает шанс на ошибки и создаёт риск блокировки учётной записи.
- Длинные пароли часто забываются и возникнет необходимость частого сброса пароля пользователям.
Также в данном поле дана рекомендация по созданию паролей через ключевые фразы, которые пользователи могли бы легче запоминать.
**Аудит**: дана общая информация по принципу проверки через визуальный осмотр соответствующего раздела
(**Remediation**): Рекомендует настроить через локальную групповую политику по следующему пути:
`
Computer Configuration\Policies\Windows Settings\Security Settings\Account Policies\Password Policy\Minimum password length
`
Также указаны обычные значения по умолчанию: 7 символов для доменных пользователей и 0 символов для автономных серверов. 
**Ссылки**: Даны ссылки на ресурсы Microsoft по парольной политике. Также указан идентификатор политики GRID: MS-00000004
**Дополнительная информация**: Дана рекомендация по использованию ключевых фраз для паролей по примеру "Я хочу выпить молочный коктейль за 5 долларов". Также уточняют, что длину пароля необходимо определять в соответствии с бизнес-требования целевой организации.
